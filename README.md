# DM System — связанный мир (локация + фракции + навыки)

## Что это

Слияние четырёх файлов в одну модульную систему:

| Файл-источник | Что из него взято |
|---|---|
| `Dm_complex_old.py` (оригинал) | Онтология 12 осей, движок навыков (target/direction/duration/trigger/mana/коллапсы), система эпох |
| `Dm_location.py` | BSP-генерация карты этажей внутри случайной "маски" формы + флейвор локации |
| `Dm_fraction.py` | Экосистема (эндемики/захватчики), дипломатия, роли мобов, тактика/эмоции |
| `Dm_skills.py` | Синергии осей (пара "сильных" осей → название феномена вроде "Некромантия") + 5-уровневая лестница текста на каждую ось |

## Структура

```
dm_system/
├── main.py                      # точка запуска — единственный файл, который дергает всё остальное
├── core/
│   ├── vector.py                 # класс Profile — 12-осевой вектор (см. ниже)
│   ├── world.py                  # Era — общая эпоха мира, роль/уровень существ от неё
│   └── schema.py                 # загрузка + валидация JSON-таблиц с понятными ошибками
├── generators/
│   ├── location_gen.py            # локация: профиль, карта этажей, сублокации
│   ├── faction_gen.py             # фракции, мобы, дипломатия
│   └── skill_gen.py               # навыки (движок + флейвор)
├── data/                          # ВСЕ таблицы и веса — редактируются без прикосновения к коду
│   ├── axes.json                   # 12 осей: описание + КТО НА КОГО ВЛИЯЕТ (correlations)
│   ├── eras.json                   # эпохи/парадигмы/титулы
│   ├── location_tables.json        # материалы/назначение/атмосфера/лут локации
│   ├── faction_tables.json         # имена/роли/иерархия/цели/дипломатия
│   └── skill_tables.json           # механизмы/синергии/лестницы текста навыков
└── saves/                          # сюда main.py --save кладёт JSON-снимки сгенерированных миров
```

## Как запустить

```bash
cd dm_system
python3 main.py                          # один случайный мир
python3 main.py --seed 42                # повторяемый мир (для отладки/показа)
python3 main.py --factions 4 --save my_world.json
```

Можно гонять генераторы по отдельности (для отладки конкретного куска):
```bash
python3 -m generators.location_gen
python3 -m generators.faction_gen
python3 -m generators.skill_gen
```

## Главная идея: `core.vector.Profile`

Раньше (в трёх аутсорс-файлах) профиль существа/локации был **плоским словарём**
`{"matter.density": 0.3, ...}` — данные, но без памяти о том, как оси связаны
между собой. В оригинале была структура `Vector3` (по одной на домен), что уже
лучше, но тоже без явных межосевых связей.

`Profile` — это словарь на 12 осей, который **знает свою собственную структуру**
из `data/axes.json`: у каждой оси есть описание, разметка "хорошо ли расти вверх"
(`good_high`) и список `correlations` — на какие другие оси и с каким весом она
слегка давит при генерации/мутации.

Это даёт то, что вы просили: **вектора влияют друг на друга**, но не жёстко и
не непредсказуемо:

1. Профиль генерируется/мутирует как обычно (независимый шум на каждую ось).
2. Затем **один раз** проходит `_propagate()`: для "сработавших" (не нулевых)
   осей соседние оси получают лёгкий толчок по весу из `axes.json`.
3. Всё клэмпится в `[-1, 1]`.

Хотите добавить новую связь ("высокая опасность локации повышает шанс редкого
лута")? Просто дописываете `correlations` в `axes.json` — код ничего не заметит,
кроме того, что профили начнут *чуть* чаще получаться такими. Если опечататься
(например, указать вес 999 или несуществующую ось) — `core/schema.py` бросит
понятную ошибку **при старте**, а не через полчаса где-то в рендере.

Наследование профиля (локация → фракция → моб) — тоже одна функция:
`Profile.spawn_from(parent, sigma)`, вместо трёх похожих кусков кода в трёх
файлах.

## Общий мир (эпоха)

`core/world.py::Era.roll()` вызывается **один раз** в `main.py` и передаётся
во все генераторы. Каждая фракция/моб роллит свою `paradigm/magic_lvl/tech_lvl`
через `roll_being_paradigm(era)` — они тяготеют к уровню эпохи (мир на 10 уровне
не рождает мага уровня 1 массово), но каждый бросок независим и случаен.
Это и есть "связанный мир, но не одинаковое развитие".

## Известные упрощения (флагую честно, как просили)

- Из оригинала **не перенесена** отдельная ветка `roll_material`/`generate_mage_vectors`
  (anchor-домен → материя → инфо → энергия по специальным формулам для "школы магии").
  Это была довольно специфичная генерация именно персонажа-мага с отдельной
  "материей школы" и "личностью мага" как двумя разными векторами. Я заменил её
  на более простую and universal `skill_gen.make_caster(profile, era, paradigm, mastery)`,
  которая работает на любом `Profile` (в том числе на мобах из `faction_gen`), но
  не воспроизводит тонкие формулы "material ↔ info ↔ energy" пайплайна оригинала.
  Если эта конкретная механика (роллинг материала школы) вам критична — скажите,
  вынесу её отдельным модулем `generators/school_gen.py` поверх того же `Profile`.
- Таблица `spacetime` в `mechanism_base_weights` (skill_tables.json) — моя
  достройка: в оригинале навыки по пространству-времени не выделялись отдельным
  доменом с весами механизмов, я добавил разумные веса по аналогии.
- Мелкий баг с двойным пробелом в описании атмосферы локации ("Наполнена  обычным
  фоном") — это из оригинального `Dm_location.py` (когда `amplitude_power` пустой
  на среднем банде), не трогал, чтобы не менять флейвор без запроса.

Если что-то из кода читается тяжело — говорите, какой файл/функция, распишу подробнее
или упрощу.

## Боевая математика: подушки безопасности (`core/cushion.py`)

Смерть/вывод моба из строя идёт НЕ через общий "хп-бар", а через 18 независимых
подушек (9 осей × 2 знака: matter.density/cohesion/plasticity, info.order/agency/
meaning_density, energy.charge/carrier/amplitude). Убить можно через любую.

**Модель общего пула.** Формулы (`base_threshold`, `self_reinforce`,
`contributors`) считают не готовый порог, а СЫРОЙ ВЕС каждой подушки. Все 18
весов нормализуются на фиксированный `total_resist_pool` (сейчас 6.3, откалиброван
как сумма нейтральных базовых весов) — архитектура вектора не наращивает
суммарную живучесть, а перераспределяет фиксированный пул. Раскачал одну
подушку — забрал вес у остальных 17.

**Асимметрия plus/minus.** Раньше contributors по умолчанию действовали на ОБЕ
стороны подушки одинаково (пересращение и распад одной и той же оси защищались
одним и тем же набором факторов — замечено как нелогичное). Теперь contributors
явно помечены `"affects": "plus"|"minus"|"both"` — например, `matter.cohesion`
и `info.order` защищают конкретно от РАССЕИВАНИЯ density (minus), а
`matter.plasticity` — конкретно от РАЗДАВЛИВАНИЯ (plus). `info.order` даже
работает в РАЗНЫЕ стороны на разных знаках `info.agency`: помогает держать волю
под давлением (minus), но усугубляет рекурсивный обвал самосознания (plus) —
осознанный дизайн-выбор, не инвариант, можно оспорить.

- Пороги считаются от СМЕЩЕНИЯ базового (природного) профиля существа, не от
  абсолютного нуля.
- Полная симметрия: у всех 9 осей в scope есть и плюс, и минус в
  `data/axes.json → collapse_effects`. Часть формулировок помечена ЧЕРНОВИК.
- `spacetime.*` вне пула — ждёт механики "разница атакующий/цель".

Живая демка (крутилка 12 осей + подушки в реальном времени, теперь синхронизирована
с моделью пула, meaning_density и асимметрией) — отдельный файл `vivisector.html`.

## Слой конвертации удара: `core/damage.py` и `core/attack.py`

Отдельный от подушек слой — не "далеко ли до слома", а "что именно происходит
при попадании".

**`damage.py`** — немонотонная (парабола) кривая для кинетики против density +
общий монотонный размер-множитель. Не откалибровано, первая демонстрация формы.

**`attack.py`** — механика "чем/как/с какой силой": `Attack(mechanism, axis,
direction, magnitude)`. Сплеш-урон по связанным осям вытащен ПРЯМО из
`cushion_table.json → contributors` (та же сеть связей, что для сопротивления,
не отдельная таблица) — сумма всех сдвигов по построению равна `magnitude`,
никогда не больше. Механизм работает как модификатор: масштабирует вес каждой
затронутой оси (включая основную) через `skill_tables.json →
mechanism_base_weights[domain]` — урон концентрируется в "родном" домене
механизма.

**Найденный побочный эффект, требует вашего решения:** если механизм
тематически НЕ связан с доменом целевой оси (например "Ментальное воздействие"
целится в `matter.density`), у него нет записи в `mechanism_base_weights` для
domain=matter (дефолт 0.3), а его контрибьютор `info.order` получает полный вес
3.0 (он в info-домене, где механизм как раз силён) — в итоге БОЛЬШАЯ часть урона
неожиданно утекает в `order`, а не остаётся на `density`. Технически это
воспроизводимо и по построению суммы честно, но по ощущениям странно: "неправильный
инструмент для оси" перенаправляет урон вместо того, чтобы просто быть слабым.
Возможно, здесь и нужна механика-гейт из вашего пункта "1" (мехранизм проверяет
применимость, а не просто масштабирует) — не стал решать это сам.

## Имена и материя: `core/names.py`, `core/inflect.py`, `data/names_tables.json`

Портировано из присланного `Dm_fraction_2.txt` — там были два больших
улучшения, которых не было в первой сборке:

- **27-комбинационная матрица материи** (`matter_matrix3` в `names_tables.json`,
  из оригинала Dm_complex_old.py) вместо бедной 5-полосной. `describe_material()`
  делает `band3(density).band3(cohesion).band3(plasticity)` lookup — даёт
  фразы вроде "неньютоновская масса — твёрдая при ударе, течёт в покое" вместо
  сухого "тяжёлый армированный композит". Особый случай "твёрдой пустоты"
  (низкая density, высокая cohesion) разворачивается по `paradigm` — маг,
  технарь и синтезист получают материал, буквально сделанный по-разному.
  Используется в материале локации/сублокаций (`get_material()` в
  `location_gen.py`), которому теперь передаётся `paradigm` (через
  `core.world.era_paradigm_to_being()` из `main.py`).
- **Грамматическое согласование рода** (`core/inflect.py::inflect_adj`) —
  эпитеты и причастия склоняются под род существительного (формы локации,
  роли мобов), а не просто конкатенируются как попало.
- **Богатые эпитеты/родительные падежи/действия по (ось, знак)**
  (`mob_epithets`/`axis_genitive`/`axis_action`, по ~12-16 вариантов на
  каждую из 24 комбинаций вместо 1-3) — имя моба теперь строится по одному
  из 5 случайных паттернов (`generate_mob_title()`), а не по одному жёсткому
  шаблону.
- `generate_location_name()` — бонус, не подключён по умолчанию в `main.py`
  (там локация всё ещё получает явное имя параметром), но доступен, если
  захотите генерировать имя локации по её профилю так же, как у фракций/мобов.

Старая `generate_mob_name()` в `faction_gen.py` оставлена, помечена
УСТАРЕЛО, никем не вызывается — можно удалить, если не нужна на память.
